How dbt Connects to Your Data Platform

From a credentials file on your laptop to a secure service account in production — understand every layer of dbt's connection model.

💡
dbt doesn't store data — it runs SQL against your warehouse. To do that, it needs to know: what platform to connect to, where on that platform, and who you are. This walkthrough covers every layer of that connection model.

Three things every dbt connection needs
# 1. WHAT — the platform type (drives which adapter is used)
type: snowflake

# 2. WHERE — account location, database, schema, and warehouse/compute
account: abc123.us-east-1
database: ANALYTICS
schema: dbt_dev
warehouse: COMPUTE_WH

# 3. WHO — authentication (covered in step 3)
user: jane@company.com
password: "{{ env_var('DBT_PASSWORD') }}"
The adapter (dbt-snowflake, dbt-bigquery, etc.) is the bridge between dbt's generic SQL compilation and each platform's specific connection protocol. You install it as a Python package alongside dbt Core, or it's bundled in dbt Cloud.
📄
profiles.yml is your local connection file. It lives at ~/.dbt/profiles.yml — outside your project directory — so credentials are never accidentally committed to git. Each profile has named outputs called targets, and target: sets the default.
📄
~/.dbt/profiles.yml — lives outside the project, never committed. Has named targets for dev and prod.
~/.dbt/profiles.ymlYAML
my_project:                           # must match "profile:" in dbt_project.yml
  target: dev                         # which output is active by default
  outputs:

    dev:                              # personal development target
      type: snowflake
      account: abc123.us-east-1
      user: jane@company.com
      password: "{{ env_var('DBT_PASSWORD') }}"  # never use a literal string here
      role: TRANSFORMER
      database: ANALYTICS_DEV         # isolated dev database
      schema: dbt_jane                # per-developer schema
      warehouse: COMPUTE_WH
      threads: 4

    prod:                             # production target
      type: snowflake
      account: abc123.us-east-1
      user: dbt_service_account       # machine identity, not a person
      private_key_path: /secrets/snowflake_key.p8  key-pair auth for prod
      role: TRANSFORMER_PROD
      database: ANALYTICS             # production database
      schema: dbt_prod
      warehouse: COMPUTE_WH_LARGE
      threads: 8
target: dev
Your personal sandbox. Writes to a dev database and your own schema. Uses your credentials.
target: ci
Short-lived CI runs. Usually a service account with a throwaway schema that's dropped after tests pass.
target: prod
Production runs via a locked-down service account. Writes to the canonical schema that downstream tools query.
dbt_project.yml — linking the project to the profile
name: my_project
version: '1.0.0'
profile: my_project   # must match the top-level key in profiles.yml

model-paths: ["models"]

BigQuery Databricks
Username + Passwordavoid in prod
  type: snowflake
  user: jane@company.com
  password: "{{ env_var('DBT_PASSWORD') }}"   # always use env_var, never a literal
  account: abc123.us-east-1
  role: TRANSFORMER
⚠️ Password auth can't enforce MFA at the service account level and is harder to rotate. Reserve it for local dev only — use key-pair or OAuth for any shared or automated environment.
Key-Pair Authenticationrecommended for prod
  type: snowflake
  user: dbt_service_account
  private_key_path: /secrets/snowflake_rsa_key.p8   # path to the private key
  private_key_passphrase: "{{ env_var('DBT_KEY_PASSPHRASE') }}"  # if key is encrypted
  account: abc123.us-east-1
  role: TRANSFORMER_PROD
✓ Generate a key pair with openssl genrsa, attach the public key to the Snowflake user via ALTER USER ... SET RSA_PUBLIC_KEY='...', and store the private key in a secrets manager. dbt Cloud handles this securely in its Connections UI.
OAuth / SSOtoken-based
# Snowflake — external OAuth or Snowflake OAuth
  type: snowflake
  authenticator: oauth
  token: "{{ env_var('DBT_OAUTH_TOKEN') }}"
  account: abc123.us-east-1

# BigQuery — Application Default Credentials (ADC)
  type: bigquery
  method: oauth
  project: my-gcp-project
  # one-time local setup: gcloud auth application-default login
  # after that, ADC handles token refresh automatically
Service Account / Tokenfor CI/CD & prod
# BigQuery — service account JSON keyfile
  type: bigquery
  method: service-account
  project: my-gcp-project
  keyfile: /secrets/bq-service-account.json    # path to the JSON key
  dataset: dbt_prod

# Databricks — personal access token (PAT)
  type: databricks
  host: abc-123.cloud.databricks.com
  http_path: /sql/1.0/warehouses/abc123
  token: "{{ env_var('DBT_DATABRICKS_TOKEN') }}"  # generate in Databricks UI under User Settings
🔌
Each platform has its own adapter and its own connection fields. Install the adapter for your warehouse (pip install dbt-snowflake, etc.) alongside dbt Core. The profiles.yml structure is the same, but the fields inside each target differ.
🔌
Same profiles.yml structure, different fields per platform. Install the right adapter and configure accordingly.

Databricksdbt-databricks
type: databricks
host: abc-123.cloud.databricks.com
http_path: /sql/1.0/warehouses/xyz
token: "{{ env_var('...') }}"
catalog: main
schema: dbt_jane
threads: 4
Redshiftdbt-redshift
type: redshift
host: abc.us-east-1.redshift.amazonaws.com
port: 5439

password: "{{ env_var('...') }}"
dbname: analytics
schema: dbt_jane

Key differences: Snowflake uses account + database + schema. BigQuery uses project + dataset (no separate database concept). Databricks uses host + http_path + catalog. Redshift uses a host hostname like a Postgres connection.

dbt Cloud replaces profiles.yml with a UI-based connection model. Credentials are stored encrypted — never on disk. Each environment (development, production) gets its own connection config, and developer credentials are set per-user in their Profile settings.
☁️
No profiles.yml in dbt Cloud. Connections live in the UI. Dev and prod use the same platform config but different credentials.

dbt Cloud environment model
Development Environment
  Connection:  Snowflake (shared — account, role, warehouse)
  Credentials: per-developer OAuth or username/password (set in Profile → Credentials)
  Schema:      dbt_<username>   ← automatically namespaced per developer

Deployment Environment (Production)
  Connection:  Snowflake (same shared config)
  Credentials: service account key-pair (set in Environment → Connection settings)
  Schema:      dbt_prod

Both environments share the same Connection object (host / account / warehouse / role).
What differs: who is authenticating, and where models are written.
✅
A few rules that prevent credential disasters and environment bleed.
✅
The connection rules that matter most in practice.
🚫
Never commit credentials to git
profiles.yml lives at ~/.dbt/, outside your project root, for exactly this reason. If you use a .env file for local variables, add it to .gitignore before anything else.
✅
Use env_var() for every secret
Wrap all secrets in {{ env_var('VAR_NAME') }}. dbt throws a clear error if the variable is missing — better than silently using a wrong or empty value.
✅
Give each developer their own schema
Set schema: dbt_{{ env_var('DBT_USER') }} in your dev target. This prevents developers from overwriting each other's models during active development.
✅
Use key-pair auth or OAuth for production
Password-based service accounts can't enforce MFA and are harder to rotate. Key-pair (Snowflake) and service accounts / ADC (BigQuery, Databricks) are the production standard.
✅
Separate dev and prod databases, not just schemas
Use ANALYTICS_DEV for development and ANALYTICS for production. This makes it physically impossible to overwrite production data with a local dbt run.
