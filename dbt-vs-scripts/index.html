This is a complete Python pipeline — the kind you'd inherit today. It builds a customer order summary using pandas. Highlighted lines are the things that create problems at scale. Read the inline labels, then move to the next tab to see how dbt solves each one.
build_customer_summary.py — full pipeline script 1 file · ~45 lines 
import pandas as pd
import sqlalchemy as sa

engine = sa.create_engine("postgresql://prod-warehouse:5432/db")  ⚠ hardcoded connection — no dev/prod separation

# Step 1: clean customers
tmp_customers = pd.read_sql("SELECT * FROM raw.customers", engine)  ⚠ hardcoded schema — rename breaks silently
tmp_customers['email'] = tmp_customers['email'].str.lower().str.strip()
tmp_customers['first_name'] = tmp_customers['first_name'].fillna('Unknown')
tmp_customers['last_name'] = tmp_customers['last_name'].fillna('Unknown')
tmp_customers['region'] = tmp_customers['region'].fillna('Unassigned').str.upper()

# Step 2: clean orders
tmp_orders = pd.read_sql("SELECT * FROM raw.orders", engine)  ⚠ hardcoded schema
tmp_orders = tmp_orders[
    tmp_orders['order_id'].notna() & (tmp_orders['status'] != 'cancelled')
][['order_id', 'customer_id', 'order_date', 'status', 'total_amount']]

# Step 3: clean payments
tmp_payments = pd.read_sql("SELECT * FROM raw.payments", engine)  ⚠ hardcoded schema
tmp_payments = tmp_payments[
    tmp_payments['payment_id'].notna() & (tmp_payments['amount'] > 0)
].rename(columns={'amount': 'payment_amount', 'created_at': 'payment_date'})

# Step 4: join orders + payments  ⚡ must wait for steps 2 & 3 — no parallelism possible
agg = tmp_payments.groupby('order_id').agg(
    total_paid=('payment_amount', 'sum'),
    payment_count=('payment_id', 'count')
).reset_index()
tmp_orders_with_payments = tmp_orders.merge(agg, on='order_id', how='left')
tmp_orders_with_payments['outstanding_balance'] = (
    tmp_orders_with_payments['total_amount']
    - tmp_orders_with_payments['total_paid'].fillna(0)
)

# Step 5: final summary
summary = tmp_customers.merge(
    tmp_orders_with_payments, on='customer_id', how='left'
)
result = summary.groupby(
    ['customer_id', 'email', 'first_name', 'last_name', 'region']
).agg(
    lifetime_orders=('order_id', 'count'),
    lifetime_revenue=('total_amount', 'sum'),
    lifetime_payments=('total_paid', 'sum'),
    most_recent_order_date=('order_date', 'max')
).reset_index()

result.to_sql('fct_customer_order_summary', engine,  ⚠ full overwrite every run — no incremental option
             schema='analytics', if_exists='replace', index=False)

# Script ends. All 5 DataFrames dropped from memory.
# If step 4 failed: nothing survives to inspect.  ⚠ debug from scratch
What you're looking at: One file, 5 logical steps, all data loaded into Python memory, hardcoded connection strings and schema names, full table overwrite on every run, and zero intermediate state when something fails. It works — until it doesn't. The next tab shows what dbt does differently.

    COALESCE(first_name, 'Unknown')  AS first_name,

In the Python world, "what is a valid order" ends up defined in five different scripts — and they drift.

SELECT
    o.order_id, o.customer_id, o.order_date, o.total_amount,

    COUNT(p.payment_id)                               AS payment_count,

-- Materialization strategy (table, incremental, view) is config, not code.
SELECT
    c.customer_id, c.email,
    c.first_name || ' ' || c.last_name AS full_name, c.region,
    COUNT(op.order_id)                  AS lifetime_orders,
    COALESCE(SUM(op.total_amount), 0)  AS lifetime_revenue,
    COALESCE(SUM(op.total_paid), 0)    AS lifetime_payments,
    MAX(op.order_date)                  AS most_recent_order_date
FROM {{ ref('stg_customers') }} c
LEFT JOIN {{ ref('int_orders_with_payments') }} op
    ON c.customer_id = op.customer_id
GROUP BY c.customer_id, c.email, full_name, c.region
You've now seen all 5 steps. The transformation logic didn't change. What changed: (1) transformations run in the warehouse, not in Python memory — no memory ceilings, no data transfer overhead; (2) each step is a real, persistent, queryable table — not a variable that vanishes; (3) if_exists='replace' is gone — materialization is a one-line config, not code you have to maintain.
The pattern: Same transformation logic throughout. What dbt changes is the execution environment and the wrapper — transformations move from Python memory into the warehouse, pd.read_sql("... FROM raw.x") becomes source(), tmp_x variable references become ref(). Those three substitutions unlock everything else.
📂
Before the lineage graph makes sense, it helps to understand what the prefixes mean and how dbt tracks where data enters and flows through the pipeline.
stg_
Staging — one file per source table. Cleans, renames, casts. Raw in, usable columns out. Nothing is joined here.
int_
Intermediate — joins and transforms that aren't the final output. Shared building blocks between mart models.
fct_ / dim_
Marts — what BI tools and business users actually query. One table per business question.

source() — where raw data enters

pd.read_sql("... FROM raw.customers", engine)→{{ source('raw', 'customers') }}
pd.read_sql("... FROM raw.orders", engine)→{{ source('raw', 'orders') }}
pd.read_sql("... FROM raw.payments", engine)→{{ source('raw', 'payments') }}
The Python script hardcoded schema names and a connection string. If a schema renames, the script throws an error — or worse, silently returns stale data. source() registers these as official entry points — dbt can test them, check freshness, and alert you before a bad data day becomes a bad dashboard day. It also resolves the correct schema per environment automatically, so dev and prod never share the same hardcoded string.

ref() — how models depend on each other

tmp_orders (Python variable)→{{ ref('stg_orders') }}
tmp_payments (Python variable)→{{ ref('stg_payments') }}
tmp_customers, tmp_orders_with_payments→ref('stg_customers'), ref('int_orders_with_payments')
Every ref() is a declared dependency. dbt reads them all and builds the execution order automatically. You don't write orchestration code — the ref() calls are the orchestration. In the Python script, the order was enforced by line numbers and comments; here it's enforced by the system.
This graph is the Python script made visible. The script had this same structure — it just wasn't visible. You had to read 45 lines of Python to understand that step 4 depended on steps 2 and 3. dbt makes it explicit, queryable, and documented automatically.

How does dbt know it can parallelize? Because stg_customers, stg_orders, and stg_payments each only reference source() — they have no ref() edges between them. dbt reads the graph, sees no dependencies, and dispatches all three at once. The ref() declarations from tab 2 are the orchestration — you didn't write any extra config to make this happen.
What about using Python with an orchestrator like Airflow? You can parallelize Python scripts with Airflow or Prefect — but now you're maintaining two systems: the transformation logic in Python, and the DAG definition in the orchestrator. In dbt, the ref() calls inside your SQL files are the DAG. There's no separate orchestration file to keep in sync.
At scale: This example has 5 models. Real projects have hundreds. A pipeline with 40 independent staging models runs in roughly the time it takes to run one, given enough threads — no orchestrator config required.
🔬
The Python script works until it doesn't. When it breaks, the debugging experience is fundamentally different. Click each scenario to compare.
Pipeline fails at the join step (step 4)▼
python script

Step 4 failed. The DataFrames from steps 1–3 were in memory — they're gone when the process died. Re-run the entire script from the top to reproduce the failure. Every debug cycle = full re-run, including all the expensive pd.read_sql() calls.

dbt

Step 4 failed. stg_orders and stg_payments are still in the warehouse as real tables. Run dbt run -s int_orders_with_payments to re-execute only that model. Upstream work stays intact.

The difference comes from one thing: dbt models are real, persistent tables in your warehouse. Python DataFrames live in memory and die with the process. That's the whole game.
Wrong row count in the final output▼
python script

Fan-out in the merge? Bad filter in step 2? You add print(df.shape) calls throughout, re-run the whole script, and stare at numbers. Or you temporarily write intermediate CSVs to disk. Multiple full re-run cycles to isolate the problem.

dbt

Query stg_orders — how many rows? Query int_orders_with_payments — how many? The difference shows exactly where the fan-out happened. No re-run needed. The intermediate tables are just sitting there in the warehouse.

Because every intermediate model is a real table, you can bisect a row count problem the same way you'd bisect a bug in code — without re-running anything.
Upstream schema change — a column got renamed▼
python script

The rename hits mid-script: KeyError: 'amount'. Steps 1 and 2 already ran and loaded data into memory. Now you're not sure what state anything is in. Find all the places in all scripts where 'amount' is referenced — there's no central list.

dbt

stg_payments is the only model that touches raw.payments. Fix the column reference there. Every downstream model gets the fix automatically on the next run. One change point, not five scripts to grep through.

The stg_ layer acts as an isolation boundary. Raw schema changes only affect one file — the staging model for that source.
3am on-call alert fires▼
python script

Script failed. No DataFrames anywhere — they were in memory. Re-run the full script to reproduce. Add print() statements, re-run again. Check if any temp CSVs were written. Multiple cycles before you even start fixing. All the expensive source queries run again each time.

dbt

dbt run logs show the exact model that failed and the exact SQL error. Intermediate models are still in the warehouse. Fix the model, run dbt run -s failed_model+ to re-run it and only its dependents.

This is where the value becomes visceral. The engineer on-call at 3am cares a lot about whether the intermediate state survived the failure.
What else depends on payments data?▼
python script

Grep the scripts folder for raw.payments and hope every script uses the same connection string format. Hope nobody copied the logic into a Jupyter notebook. No system-of-record for what reads what.

dbt

The lineage graph shows every downstream model that references stg_payments. Click on it in the dbt DAG and see the full impact of any change instantly. A new engineer can answer "what depends on this?" in 5 seconds.

A grep of the codebase finds what you know to look for. The dbt DAG shows you what you didn't know existed.
The root cause of every dbt debugging advantage: Intermediate models are real, persistent, queryable tables. Python DataFrames live in memory and disappear when the process ends. Every other difference flows from that one fact.
